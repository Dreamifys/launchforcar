# 国内车机应用技术架构地图

## 1. 文档目标

本文用于分析国内常见车机版应用的实现形态，帮助本项目在设计桌面联动、默认启动、分屏调起、媒体控制和适配策略时建立现实基线。

本文仅基于公开资料、公开下载信息、产品能力描述与 Android 车机开发常识进行归纳，不涉及逆向与非公开实现细节。

## 2. 核心结论

国内市面上的车机版应用，绝大多数底座仍然是 Android APK，但实现路线明显分为三类：

1. 导航类：原生 Android 壳层 + 大量 NDK 能力，属于重原生架构。
2. 音乐类：原生媒体框架 + 播放内核 + 动态业务层，属于原生媒体架构。
3. 内容聚合类：原生容器 + 播放能力 + 配置化内容流，属于内容平台架构。

它们通常不是 Google 标准模板化 Android for Cars 应用，而是直接运行在 OEM 自定义 Android 或 AAOS 改版系统上的大屏 Android 应用，并针对横屏、方控、语音、停车态和分辨率碎片化做深度适配。

## 3. 为什么国内车机应用普遍不是模板化 Car App

Google 官方 Android for Cars 更强调模板、低分心和受限类别，适用于导航、媒体、天气、IoT 等特定场景。

但国内量产车机和改装车机更常见的是以下现实约束：

1. 系统碎片化严重，Android 版本跨度大，甚至仍存在 Android 4.4 到 Android 10 的大量存量设备。
2. OEM 系统权限、系统白名单、分屏规则、前台保活策略差异很大。
3. 许多应用需要完整掌控横屏大屏 UI，而不是受模板约束。
4. 内容、活动、推荐和商业化频繁调整，业务层需要高动态化能力。
5. 地图、媒体、语音和视频对性能要求高，核心能力必须牢牢掌握在原生层。

所以国内所谓“车机版应用”，本质上更接近“面向车载座舱定制的大屏 Android 客户端”，而不是标准 Android Auto 模板应用。

## 4. 主流技术架构分层

从产品实现角度，国内车机应用常见为四层结构：

### 4.1 系统适配层

负责处理与车机环境直接相关的差异：

1. 横屏、大屏、异形屏和低分辨率适配
2. 方向盘按键、旋钮、实体键事件适配
3. 语音唤醒、系统广播、默认桌面、开机自启
4. 分屏、多窗口、停车态、驾驶态限制
5. 厂商白名单、保活、电源策略、权限弹窗差异

这一层通常只能由原生 Android 完成。

### 4.2 原生能力层

负责真正影响稳定性和性能的核心能力：

1. 地图渲染、导航、定位、TTS
2. 音频焦点、后台播放、蓝牙、MediaSession
3. 视频播放、解码、缓存、字幕、投屏
4. 登录态、账号安全、数据缓存、网络重试

这一层通常使用 Java 或 Kotlin 结合 C 或 C++，其中地图和复杂多媒体能力往往明显依赖 NDK。

### 4.3 业务容器层

负责用户可见的业务界面与流程：

1. 首页、频道页、搜索页、播放页
2. 推荐流、活动页、运营位、专题页
3. 账号引导、会员、权益、弹层和任务系统
4. 场景卡片与桌面快捷入口

这一层通常不是完全静态代码，而会混合组件化、动态配置和远程实验开关。

### 4.4 云端中台层

负责让车机客户端“持续变化”：

1. 内容分发和推荐策略
2. 远程配置、AB 实验、灰度开关
3. 车型差异化策略
4. 埋点、画像、商业化和广告控制

因此，许多车机应用真正的“产品能力”并不只在 APK 本身，而在客户端与中台协同。

## 5. 典型类型分析

### 5.1 导航类应用

代表：高德地图车机版、百度地图车机版

#### 高概率架构

1. Java 或 Kotlin 负责应用壳、页面流转、权限与系统交互。
2. 地图渲染、路径规划、定位融合、导航引擎、语音播报等关键能力在 C 或 C++ 层实现。
3. 与系统语音、蓝牙、车机分屏和前后台切换存在较深耦合。
4. 为兼容大量存量车机，会保留对旧 ABI 和旧 Android 版本的支持。

#### 判断依据

1. 导航类应用需要高性能地图渲染和复杂算法，不适合纯 WebView 方案。
2. 你当前使用的高德车机 APK 仅包含 armeabi-v7a so，这说明它明显包含原生能力层，并不是轻量 H5 应用。
3. 此类应用通常强依赖 Intent 协议、系统定位、语音播报和前台服务。

#### 导航类对桌面的启发

1. 不要试图“嵌入式重写”导航能力，应以调起、分屏、状态联动为主。
2. 需要预留包名、启动 Activity、深链 Intent 和异常回退策略。
3. 需要允许不同地图应用有不同适配表，而不是假设统一接口。

### 5.2 音乐类应用

代表：网易云音乐车机版、QQ 音乐车机版、酷我音乐车机版

#### 音乐类高概率架构

1. 原生 Android UI 负责主页、播放页、歌词页和账号流。
2. 底层依赖 Android 媒体框架、MediaSession、AudioFocus 和蓝牙控制。
3. 播放内核基于系统解码能力，并可能组合 ExoPlayer、FFmpeg、ijkplayer 一类组件。
4. 推荐页、活动位、专题页和部分运营模块使用动态配置甚至 H5 承载。

#### 音乐类公开信号

1. 网易云音乐车机版已由官方单独提供 APK 分发，并公开强调车机适配、品牌合作和可独立安装。
2. 车机版功能普遍强调方向盘切歌、后台播放、账号同步和大屏简化 UI。
3. 这些能力都要求其核心播放和控制逻辑运行在原生层，而非单纯 Web 内容页。

#### 音乐类对桌面的启发

1. 与音乐应用的联动优先使用 MediaSession 和系统媒体控制，而不是依赖页面抓取。
2. 桌面应支持显示媒体状态、封面、播放控制和恢复播放。
3. 应为不同音乐应用保留独立的兼容入口和包名白名单。

### 5.3 内容聚合类应用

代表：车鱼视听、车载短视频/视频聚合类应用

#### 内容类高概率架构

1. 原生 Android 外壳负责大屏适配、语音、方控和系统接入。
2. 音视频播放能力仍依赖原生播放器内核。
3. 推荐流、频道页、运营位和内容组织方式更偏配置化和平台化。
4. 某些内容模块可能通过 H5 或动态卡片承载，以便快速上线活动和专题。

#### 内容类公开信号

1. 车鱼视听被公开描述为聚合抖音短视频、西瓜视频、汽水音乐等内容源的车载产品。
2. 这类产品天然需要强推荐、强运营、强内容配置，不太可能全部用纯静态原生页面硬编码。
3. 但它同时又要求在车机环境下支持横竖屏、语音和播放体验，因此也不可能是纯 WebView 外壳。

#### 内容类对桌面的启发

1. 不要假设内容应用存在稳定的深链接规范，很多入口可能会频繁变化。
2. 更现实的联动方式是应用级调起、最近任务恢复和媒体会话读取。
3. 对内容类应用要把适配重点放在“启动成功率”和“回桌面恢复路径”，而不是内部页面级联动。

### 5.4 浏览器与工具类应用

代表：车机浏览器、文件管理器、视频播放器、投屏工具

#### 常见架构

1. 浏览器类更多依赖系统 WebView 或 Chromium 内核。
2. 文件管理与本地播放器更偏纯原生工具应用。
3. 投屏和互联类应用通常依赖较强的系统服务、网络发现和编解码能力。

#### 工具类对桌面的启发

1. 工具类适合做固定快捷位，不适合做复杂卡片联动。
2. 浏览器类可以联动网址或 Intent，但稳定性不如导航和音乐类。

## 6. 不是主流的几条路线

在国内车机应用里，以下方案存在，但通常不是主导底座：

### 6.1 纯 WebView 应用

优点是开发快，但缺点很明显：

1. 音视频和地图性能不足
2. 系统交互能力弱
3. 离线能力和稳定性差
4. 难做好车机级输入与焦点管理

因此通常只会用于活动页、会员页、运营页、帮助页，而不是整个车机应用的核心框架。

### 6.2 纯 Flutter 或纯跨平台 UI 主导

这类方案在手机业务中很常见，但在车机场景里容易碰到：

1. 老版本 Android 兼容压力大
2. 系统交互与多窗口定制度高
3. 方向盘、MediaSession、前台服务等能力仍要大量落回原生层

所以即便使用 Flutter，也更可能只是部分界面层，而不是整个产品的全部架构底座。

### 6.3 纯模板化 Android for Cars App

这类方案适合上 Play 生态，但对国内 OEM 自由度较高的座舱环境并不够用。国内很多车机版应用要做的事情远超模板允许范围。

## 7. 对 Car AI Launcher 的具体影响

本项目在设计第三方应用联动时，建议默认外部应用是“黑盒原生应用”，并据此约束桌面架构：

### 7.1 联动策略应以系统级能力为主

优先级建议如下：

1. 包名调起
2. 指定 Activity 调起
3. 深链 Intent 调起
4. 最近任务恢复
5. MediaSession 状态读取

### 7.2 不要假设第三方应用遵守统一标准

需要接受这些现实：

1. 包名会变
2. 启动页会变
3. 是否支持分屏会变
4. 是否暴露媒体会话会变
5. 不同车型上的权限行为会变

因此，桌面需要维护应用适配表，而不是只写死一套逻辑。

### 7.3 必须优先兼容老 ABI 与旧系统

你当前遇到的高德车机版仅带 armeabi-v7a，就是一个直接例子。这说明国内车机生态中仍然存在大量面向旧设备和旧兼容层的应用分发现实。

### 7.4 最有价值的桌面能力不是“嵌入第三方页面”

真正有价值的是：

1. 快速调起
2. 回桌面恢复
3. 跨应用状态感知
4. 媒体和导航的统一控制入口
5. 针对高频应用做稳定适配

## 8. 适配优先级建议

如果后续要做第三方联动适配，建议优先顺序如下：

1. 导航：高德、百度
2. 音乐：网易云、QQ 音乐、酷我
3. 视频与内容：车鱼视听、B 站车机版、浏览器视频方案
4. 工具：浏览器、文件管理器、投屏工具

原因很简单：导航和音乐最稳定、最高频，也最容易形成桌面产品差异化。

## 9. 小结

国内车机应用的主流架构，不是“一个统一的车机模板生态”，而是“原生 Android 为底、系统深适配为壳、媒体与地图能力原生化、业务层动态化”的混合架构。

这对本项目意味着两件事：

1. Car AI Launcher 必须把第三方应用当作异构黑盒来适配。
2. 联动能力要优先建设系统级调起、媒体状态和应用适配表，而不是假设车机生态会自动标准化。
